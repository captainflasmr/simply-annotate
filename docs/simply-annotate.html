1 Introduction ¶

Simply Annotate is a lightweight annotation system for Emacs that allows you to add persistent notes to any text file, or specialized non-file buffers like Info-mode manuals, without modifying the original content. It’s perfect for code review, research, note-taking, and documentation and includes threading and collaboration features.

  • Key Features

1.1 Key Features ¶

  • Non-intrusive: Annotations don’t modify your original files
  • Persistent: Notes survive between Emacs sessions
  • Threading: Add replies to annotations for conversations
  • Support for Non-File Buffers: Annotate Info manuals or other specialized buffers natively
  • Status tracking: Manage annotation lifecycle (open → resolved)
  • Multi-author support: Collaborate with team members
  • Priority levels: Organize by importance (low, normal, high, critical)
  • Tag system: Categorize with common keywords (review, bug, question)
  • Raw sexp editing: Direct access to annotation data structures for power users
  • Annotation levels: Three-tier annotation system (file, defun, line) for progressive code exploration
  • Multiple display styles: Choose between highlighting, tint, fringe, fringe bracket, both, or subtle
  • Smart action: Context-aware command that adapts to the situation
  • Rich editing: Full-featured annotation buffer for longer notes
  • Navigation: Jump between annotations with simple keybindings
  • Overview: List all annotations in an org-mode buffer with folding
  • Cross-file: Browse all annotations across files in a global org-mode overview
  • Org-mode export: Convert annotation threads to structured TODO items
  • Header-line integration: Shows annotation count and available commands
  • Dired integration: Fringe indicators in dired for files with annotations
  • Kanban board: Visual board view of project annotations grouped by status columns
  • Cross-project overview: Discover all projects with annotations (from both the global database and project.el’s known projects) in a sortable summary table with per-status counts and one-keystroke drill-down to any project’s table, org listing, or kanban board

2 Installation ¶

Requires Emacs 27.2 or later. No external dependencies.

  • Using MELPA (Recommended)
  • Keymap Configuration
  • Manual Installation

2.1 Using MELPA (Recommended) ¶

The easiest way to install Simply Annotate is through MELPA: 

(use-package simply-annotate
  :ensure t
  :demand t
  :hook (find-file-hook . simply-annotate-mode)
  :config
  (global-set-key (kbd "M-s") simply-annotate-command-map))

2.2 Keymap Configuration ¶

Simply Annotate provides two keymaps:

simply-annotate-mode-map

The minor mode keymap, intentionally minimal. Only binds M-n (next annotation) and M-p (previous annotation) so it won’t conflict with other packages.

simply-annotate-command-map

The main command keymap with short single-key bindings (j, r, n, e, etc.). You bind this to a prefix key of your choice.

The recommended prefix is M-s, which gives you M-s j (smart action), M-s r (reply), M-s l (list), M-s C (clear), etc. All keybindings in this manual assume the M-s prefix.

Important: Use :demand t with global-set-key in :config for the M-s prefix. :demand t loads the package immediately so the prefix is available everywhere in Emacs from startup.

Do not use :bind-keymap for M-s. :bind-keymap defers loading by creating an autoload proxy — a temporary non-prefix command that loads the package on first keypress. This works well for free keys like C-c a, but M-s is already Emacs’s search-map prefix. The proxy replaces search-map with a non-prefix command, so any code that subsequently tries to bind under M-s will error. Using global-set-key in :config avoids this by binding directly to the real keymap after the package has loaded.

If M-s conflicts with other packages (e.g. Consult), use :bind-keymap with a free prefix like C-c a: 

;; C-c a prefix --- :bind-keymap works well here because
;; C-c a is a free key with no existing prefix map, and
;; it defers loading until first use.
(use-package simply-annotate
  :bind-keymap ("C-c a" . simply-annotate-command-map)
  :hook (find-file-hook . simply-annotate-mode))

2.3 Manual Installation ¶

If you prefer to install manually:

  1. Download simply-annotate.el
  2. Add to your load path and bind the command map: 
    (add-to-list 'load-path "/path/to/simply-annotate")
(require 'simply-annotate)
(global-set-key (kbd "C-c a") simply-annotate-command-map)
(add-hook 'find-file-hook #'simply-annotate-mode)
(add-hook 'dired-mode-hook #'simply-annotate-dired-mode)

3 Quick Start ¶

Here’s how to get started with Simply Annotate:

  1. Open any file in Emacs
  2. Enable annotation mode: M-x simply-annotate-mode
  3. Select some text and press M-s j to create your first annotation
  4. Type your annotation text and press C-c C-c to save
  5. Navigate with M-n (next) and M-p (previous)
  6. Press M-s ' to cycle through different display styles
  7. Press M-s [ and M-s ] to cycle through annotation levels
  8. Press M-s r to add a reply to any annotation
  9. Press M-s s to set status (open, resolved, etc.)
  10. Press M-s e to edit annotation data structure directly
  11. Press M-s o to export annotations to org-mode
  12. Press M-s A to see every project that has annotations in a sortable overview

4.1 Enabling Annotation Mode ¶

To start using annotations in a buffer, enable the minor mode: 

M-x simply-annotate-mode

You can also bind this to a convenient key: 

(global-set-key (kbd "C-c A") 'simply-annotate-mode)

4.2 Creating Annotations ¶

  • Smart Action Command
  • Traditional Method
  • Multi-line Annotations

4.2.1 Smart Action Command ¶

The M-s j command (simply-annotate-smart-action) is context-aware and behaves differently depending on the situation:

With region selected

Creates new annotation or edits existing one at the selected text

On annotated text (no region)

Toggles annotation buffer visibility to show/hide the annotation

With prefix argument (C-u M-s j)

Forces edit mode on existing annotation

Elsewhere (no region, no annotation)

Creates annotation for the current line

4.2.2 Traditional Method ¶

For more explicit control:

  1. Select the region you want to annotate
  2. Press M-s j
  3. Enter your annotation text in the dedicated buffer
  4. Save with C-c C-c

4.2.3 Multi-line Annotations ¶

The annotation buffer supports full multi-line editing:

  1. Select the region you want to annotate
  2. Press M-s j
  3. Use the full annotation buffer for detailed notes
  4. Format your text as needed (multiple paragraphs, bullet points, etc.)
  5. Save with C-c C-c

4.3 Viewing Annotations ¶

When simply-annotate-mode is active:

  • Annotated text is displayed according to your chosen display style
  • The header line shows annotation count and available commands
  • Thread status and comment count displayed in header (e.g., [OPEN/HIGH:3])
  • Moving to annotated text shows annotation details in the header
  • Press M-s j on annotated text to view/edit in detail

4.4 Editing Annotations ¶

To edit an existing annotation:

  1. Place cursor on annotated text
  2. Press M-s j to open the annotation buffer
  3. Make your changes directly (the buffer starts in edit mode)
  4. Save with C-c C-c or cancel with C-c C-k

4.5 Deleting Annotations ¶

To remove an annotation:

  1. Place cursor on annotated text
  2. Press M-s - to remove the annotation

Alternatively, to remove all annotations from the current buffer at once:

  1. Press M-s C
  2. Confirm the deletion at the prompt (y or n)

Alternatively, you can edit the annotation and delete all text, then save with C-c C-c to remove it.

5 Advanced Editing ¶

  • Raw Sexp Editing

5.1 Raw Sexp Editing ¶

For complete control over annotation data structures, Simply Annotate provides raw sexp editing capabilities. This allows power users to directly manipulate the underlying Elisp data structure that stores annotation information.

  • Entering Sexp Edit Mode
  • Sexp Structure
  • Required Fields
  • Comment Structure
  • Sexp Editing Features
  • Use Cases for Sexp Editing
  • Safety and Best Practices

5.1.1 Entering Sexp Edit Mode ¶

To edit an annotation’s raw data structure:

  1. Place cursor on any annotation
  2. Press M-s e to open the raw sexp editor
  3. Edit the Elisp data structure directly
  4. Save with C-c C-c or cancel with C-c C-k

5.1.2 Sexp Structure ¶

Annotations are stored as Elisp association lists with the following structure: 

((id . "thread-123456")
 (created . "2025-06-29T10:30:00")
 (status . "open")
 (priority . "high")
 (tags . ("bug" "critical"))
 (comments . (((id . "c-1719654600-000001")
               (parent-id . nil)
               (author . "John Doe")
               (timestamp . "2025-06-29T10:30:00")
               (text . "Found a critical bug here")
               (type . "comment"))
              ((id . "c-1719657300-000002")
               (parent-id . "c-1719654600-000001")
               (author . "Jane Smith")
               (timestamp . "2025-06-29T11:15:00")
               (text . "I can reproduce this issue")
               (type . "reply")))))

5.1.3 Required Fields ¶

id ¶

Unique identifier for the thread (string)

created ¶

ISO timestamp when thread was created

status ¶

Current status ("open", "in-progress", "resolved", "closed")

priority ¶

Priority level ("low", "normal", "high", "critical")

tags ¶

List of tag strings (can be empty list)

comments ¶

List of comment/reply objects

5.1.4 Comment Structure ¶

Each comment in the comments list must have:

id ¶

Unique identifier for the comment (string, e.g. "c-1719654600-000001"). Auto-assigned for legacy comments on first access.

parent-id ¶

The id of the parent comment this replies to, or nil for root comments. This field encodes the nested reply hierarchy.

author ¶

Author name (string)

timestamp ¶

ISO timestamp when comment was created

text ¶

The actual comment text content

type ¶

Either "comment" (original) or "reply"

5.1.5 Sexp Editing Features ¶

The sexp editor provides:

  • Syntax highlighting: Key fields and values are highlighted
  • Helpful comments: Guidance about structure and valid values
  • Validation: Ensures required fields are present when saving
  • Error handling: Clear error messages for invalid structures
  • Auto-completion: Missing fields are added with defaults

5.1.6 Use Cases for Sexp Editing ¶

  • Bulk operations: Modify multiple comments at once
  • Data migration: Copy annotation structures between files
  • Precision control: Manually adjust timestamps or IDs
  • Complex threading: Create reply structures not possible through UI
  • Metadata manipulation: Bulk change authors, statuses, or priorities
  • Import/export: Transform annotations for external tools

5.1.7 Safety and Best Practices ¶

  • Always use C-c C-k to cancel if unsure about changes
  • Ensure proper Elisp syntax with balanced parentheses
  • Use ISO timestamp format: "2025-06-29T10:30:00"
  • Keep type field as "comment" or "reply"
  • Preserve required fields (id, comments, etc.)
  • Test complex changes on a backup first

6 Threading and Collaboration ¶

Lets create a thread!, reply and add in authors

  • Threading System

6.1 Threading System ¶

Each annotation can become a threaded conversation with nested replies. This is perfect for code reviews, collaborative editing, and ongoing discussions about specific parts of your code or documents.

  • Adding Replies
  • Thread Display
  • Database Migration

6.1.1 Adding Replies ¶

To add a reply to any annotation:

  1. Place cursor on annotated text
  2. Press M-s r
  3. If the thread has multiple comments, select which comment to reply to from the hierarchical completing-read menu
  4. Enter your reply text
  5. The reply is nested under the selected comment

Replies support arbitrary nesting depth. When the thread has only one comment, the reply targets it directly with no extra prompt.

6.1.2 Thread Display ¶

Threaded annotations are displayed with hierarchical indentation: 

┌─ ✎ [OPEN/HIGH] review, urgent ────────────
│ John Doe (06/19 09:49):
│  This needs review before release
│   ↳ Jane Smith (06/19 10:15):
│     I'll check this today
│     ↳ Bob Wilson (06/19 10:17):
│       Ok!
│       ↳ Jane Smith (06/19 18:17):
│         Done!
│   ↳ Bob Wilson (06/19 11:30):
│     Looks good to me, approved
└──────────────────────────────────────────┘

Each depth level adds 2 spaces of indentation. The same hierarchical rendering is used in inline overlays, the listing buffer (nested org headings), and org export.

6.1.3 Database Migration ¶

Existing annotations from before version 0.9.0 use a flat comment list without id or parent-id fields. These are automatically migrated on first access: replies (comments with type = "reply") become children of the root comment.

To migrate the entire database at once, run: 

M-x simply-annotate-migrate-threading

This is safe to run multiple times and only modifies threads that need updating.

6.2 Status Management ¶

  • Setting Status
  • Setting Priority
  • Adding Tags
  • Removing Tags
  • Searching by Tag

6.2.1 Setting Status ¶

Track the lifecycle of your annotations:

  1. Place cursor on annotated text
  2. Press M-s s to set status
  3. Choose from: open, in-progress, resolved, closed

The default statuses can be customized via simply-annotate-thread-statuses.

6.2.2 Setting Priority ¶

Organize annotations by importance:

  1. Place cursor on annotated text
  2. Press M-s p to set priority
  3. Choose from: low, normal, high, critical

6.2.3 Adding Tags ¶

Categorize your annotations with tags:

  1. Place cursor on annotated text
  2. Press M-s t to add a tag
  3. Choose from existing tags or type a new one (e.g. review, bug, question). Existing tags from the database are offered as completion candidates, but free-form input is accepted.
  4. Tags appear as review bug in the annotation display

6.2.4 Removing Tags ¶

To remove a tag from an annotation:

  1. Place cursor on annotated text
  2. Press M-s C-t to remove a tag
  3. Select from the annotation’s current tags via completing-read

6.2.5 Searching by Tag ¶

Press M-s F (simply-annotate-search-by-tag) to find annotations by tag. The command uses a two-stage workflow:

  1. Select a tag from all tags in the current scope via completing-read
  2. Select a matching annotation from the filtered results
  3. Jump to the annotation’s source file and position

The scope is controlled by prefix arguments:

M-s F

Project files only (or all files when not in a project). When called from a dired buffer under a project subdirectory, candidates are automatically narrowed to that subdirectory.

C-u M-s F

Whole project, ignoring any dired auto-narrow.

C-u C-u M-s F

Every annotated file in the database.

6.3 Multi-Author Support ¶

  • Author Configuration
  • Prompting Modes
  • Author Management Commands

6.3.1 Author Configuration ¶

Configure team members and prompting behavior: 

;; Single user (default behavior)
(setq simply-annotate-prompt-for-author nil)

;; Team collaboration
(setq simply-annotate-author-list '("John Doe" "Jane Smith" "Bob Wilson"))
(setq simply-annotate-prompt-for-author 'threads-only)  ; Prompt only for replies
(setq simply-annotate-remember-author-per-file t)       ; Remember per file

6.3.2 Prompting Modes ¶

nil

Never prompt (single-user mode) - default behavior

'first-only

Prompt once per session, then remember choice

'always

Prompt for every annotation and reply

'threads-only

Prompt only when adding replies (great for code reviews)

6.3.3 Author Management Commands ¶

M-s a

simply-annotate-change-annotation-author - Change author of existing annotation/comment

6.4 Collaboration Workflows ¶

  • Code Review Workflow
  • Document Collaboration

6.4.1 Code Review Workflow ¶

  1. Reviewer creates annotations on problematic code sections
  2. Set priority with M-s p (high for blocking issues)
  3. Add relevant tags with M-s t (#security, #performance, #style)
  4. Developer adds replies with M-s r explaining or asking questions
  5. Reviewer responds with additional context or approval
  6. Mark as resolved with M-s s when fixed
  7. Export final review with M-s o to org-mode for documentation

6.4.2 Document Collaboration ¶

  1. Author writes draft document
  2. Reviewers add annotations with feedback using different author names
  3. Track status of each feedback item with M-s s
  4. Use M-s l to get overview of all feedback
  5. Export to org-mode with M-s o for final review meeting
  6. Archive resolved items, keep open ones visible

7 Display Styles ¶

Simply Annotate supports five display styles that can be used individually or combined as a list. Styles are processed in order, so later face-based styles override earlier ones while fringe styles are independent.

  • Available Styles
  • Combining Styles
  • Changing Display Styles
  • Style Recommendations
  • Fringe Customization
  • Inline Display

7.1 Available Styles ¶

highlight

Traditional background highlighting of annotated text (good for emphasis)

tint

Subtle background colour derived from the current theme, lightened by simply-annotate-tint-amount (default 5%). Adapts automatically to any theme.

fringe

Shows small indicators in the left fringe (minimal visual impact)

fringe-bracket

Draws a vertical bracket in the left fringe spanning the full annotated region. Uses the configured fringe indicator at the top with a continuous vertical bar and a horizontal cap at the bottom. Customisable via simply-annotate-fringe-bracket-face.

bar

Draws a solid heavy vertical bar (┃ ) in the left margin of every annotated line.

bracket

Outlines annotated regions with heavy box-drawing corners (┏ , ┃ , ┗ ) for multi-line annotations, or a single ┠ indicator for single-line annotations. This is now the default display style.

subtle

Uses overline and underline to bracket annotated regions (unobtrusive)

7.2 Combining Styles ¶

Styles can be combined by setting simply-annotate-display-style to a list: 

(setq simply-annotate-display-style '(highlight fringe))
(setq simply-annotate-display-style '(tint fringe-bracket))

Additionally, inline display can be toggled on top of any style (see Display Styles).

7.3 Changing Display Styles ¶

You can change the display style in several ways:

  • Press M-s ' to cycle through all three styles
  • Set simply-annotate-display-style in your configuration
  • Use simply-annotate-update-display-style to refresh existing annotations

7.4 Style Recommendations ¶

Code Review

Use fringe style to avoid visual clutter while coding

Research and Documentation

Use highlight style for emphasis on important passages

Critical Files

Use both style for maximum visibility of important annotations

Code Exploration

Toggle inline display (M-s /) to read annotations in context without opening the annotation buffer

Threading Display

All styles work well with threaded annotations and status indicators

7.5 Fringe Customization ¶

The fringe display can be customized:

  • Choose different symbols: left-triangle, right-triangle, filled-rectangle, or custom
  • Customize colors with simply-annotate-fringe-face
  • Define custom bitmaps for unique indicators

7.6 Inline Display ¶

Inline display is an overlay that can be toggled on top of any display style with M-s / (simply-annotate-toggle-inline). When active, it renders annotation text as a box-drawn block relative to the annotated region.

The position is controlled by simply-annotate-inline-position:

after

(Default) The block appears after the annotated region: 

(defun calculate-total (items)
  ┌─ ✎ [OPEN/NORMAL] ────────
  │ Sums all item prices.
  │ Returns zero for empty list.
  └────────────────────────────
  (apply #'+ (mapcar #'price items)))
above

The block appears above the annotated region: 

  ┌─ ✎ [OPEN/NORMAL] ────────
  │ Sums all item prices.
  │ Returns zero for empty list.
  └────────────────────────────
(defun calculate-total (items)
  (apply #'+ (mapcar #'price items)))

Both inline blocks and the underlying display style (highlight, fringe, both, or subtle) are filtered to the active annotation level — cycling levels with M-s [ updates which annotations are visible.

Long annotations are truncated after simply-annotate-inline-max-lines lines (default 20). Thread metadata (status, priority, comment count) appears in the header.

  • Inline Pointer
  • Inline Customisation

7.6.1 Inline Pointer ¶

A pointer connects each inline box to the annotated text, making the association visually clear. The pointer string is indented to the annotation start column. Strings can be multiline (use \n). Set to nil or an empty string to disable.

simply-annotate-inline-pointer-after

Pointer shown between annotated text and the box below. Default: "▴"

simply-annotate-inline-pointer-above

Pointer shown between the box above and annotated text. Default: "▾"

Example pointer styles (see the Commentary section for a full list): 

;; Simple arrows (default)
(setq simply-annotate-inline-pointer-after "▴")
(setq simply-annotate-inline-pointer-above "▾")

;; Heavy L-bracket
(setq simply-annotate-inline-pointer-after "┃\n┗━▶")

8 Annotation Levels ¶

Simply Annotate provides a three-tier annotation level system for progressive code exploration. Each annotation is tagged with a level when created, and you can cycle through levels to focus on annotations at a particular granularity.

  • Level Overview
  • How Levels Work
  • Cycling Levels
  • Direct Level Selection
  • Creating Annotations at a Level
  • Backward Compatibility
  • Use Cases

8.1 Level Overview ¶

file

Whole-file descriptions. Use this level for high-level summaries of what a file does, its purpose in the project, or overall architecture notes.

defun

Function or element descriptions. The default level, suitable for documenting individual functions, classes, or major code blocks.

line

Individual syntactical elements. Use this level for fine-grained notes about specific expressions, variables, or implementation details.

8.2 How Levels Work ¶

Only annotations matching the active level are styled. Annotations at other levels have their visual markers (highlights, fringe indicators, inline blocks) removed, but the underlying text remains visible:

  • Display: Only overlays matching the active level show highlights, fringe indicators, and inline blocks.
  • Navigation: M-n and M-p visit only annotations at the active level.
  • Editing: M-s j on annotated text only opens annotations matching the active level.
  • Header display: The annotation count and position indicator reflect only the active level.

8.3 Cycling Levels ¶

Use M-s ] to cycle forward and M-s [ to cycle backward through levels: 

M-s ]: file ⇒ defun ⇒ line ⇒ file ⇒ ...
M-s [: line ⇒ defun ⇒ file ⇒ line ⇒ ...

The header-line shows annotation counts for every level, with the active level in bold (e.g. FILE:2 | DEFUN:5 | LINE:3).

8.4 Direct Level Selection ¶

Use M-x simply-annotate-set-level to jump directly to a specific level via completing-read.

8.5 Creating Annotations at a Level ¶

New annotations inherit the currently active level. For example, if you cycle to file level and then create an annotation, it will be tagged as a file-level annotation.

8.6 Backward Compatibility ¶

Existing annotations created before the levels feature was introduced automatically default to the defun level. No database migration is required.

8.7 Use Cases ¶

  • Code Exploration
  • LLM-Generated Documentation

8.7.1 Code Exploration ¶

  1. Start at file level to read high-level descriptions of what each file does
  2. Switch to defun level to understand individual functions
  3. Drill down to line level for implementation details

8.7.2 LLM-Generated Documentation ¶

Annotation levels provide a natural structure for AI-generated code descriptions:

  • Generate file-level summaries for project overview
  • Generate defun-level documentation for API reference
  • Generate line-level annotations for detailed code walkthroughs

Users can then browse these layered annotations at their preferred depth.

11 Project-Aware Annotations ¶

By default, all annotations are stored in a single global database file. Version 1.0.0 adds project-aware storage and project-scoped views, using Emacs built-in project.el.

  • Database Strategy
  • Migrating to a Project Database
  • Project-Scoped Views
  • Stale Annotation Detection
  • Position Tracking

11.1 Database Strategy ¶

The simply-annotate-database-strategy variable controls where annotations are stored:

global ¶

Single database at simply-annotate-file (default, existing behaviour).

project ¶

Per-project database at the project root, falling back to the global database for files not inside a recognised project. File keys are stored as project-relative paths, making the database portable and committable.

both ¶

Write to the project database when inside a project; read from both project and global databases, with project entries taking precedence on conflicts. Annotations from the global database migrate to the project database when their files are next saved.

The simply-annotate-project-file variable (default .simply-annotations.el) sets the filename placed at the project root.

  • Setup

11.1.1 Setup ¶

;; Store annotations at the project root
(setq simply-annotate-database-strategy 'project)

;; Or merge project and global databases
(setq simply-annotate-database-strategy 'both)

To share annotations with your team, commit .simply-annotations.el. To keep annotations private, add it to .gitignore.

11.2 Migrating to a Project Database ¶

When switching from global to project strategy, existing annotations live in the global database with absolute file paths. The migration command copies matching annotations into a project-local database with relative keys: 

M-x simply-annotate-migrate-to-project

This command:

  1. Reads the global database
  2. Finds entries whose file paths fall under the current project root
  3. Writes them to the project database with relative keys
  4. Removes migrated entries from the global database

You must have a file open inside the target project before running this command, so that project-current can detect the project root.

11.3 Project-Scoped Views ¶

simply-annotate-show-project (P in the command map) shows an org-mode listing of all annotations in the current project, grouped by directory and file. Like simply-annotate-show-all but filtered to the current project.

simply-annotate-list-project-table (C-t in the command map) opens a sortable table spanning all project files, with columns for File, Level, Stale, Line, Status, Priority, Comments, Tags, Author, and Comment.

simply-annotate-jump-to-file (f in the command map) defaults to project files when using project or both strategy. Use a prefix argument (C-u f) to show all files across all databases.

  • Automatic Directory Narrowing

11.3.1 Automatic Directory Narrowing ¶

When any project-level command (Kanban, Table, Org listing, File jump, Search by tag) is invoked from a buffer that belongs to a project subdirectory, the view is automatically narrowed to that subdirectory. This allows for a “local peek” of relevant annotations without needing to manually filter.

This auto-narrowing behavior works from both Dired buffers and regular file buffers.

Use a single prefix argument (C-u) when invoking any of these commands to force the whole project view, bypassing the automatic directory filter.

11.4 Stale Annotation Detection ¶

When the text under an annotation changes (refactoring, edits, or merges), the annotation may no longer apply to the correct content. Simply Annotate detects this by storing a content hash of the annotated text region.

  • How It Works
  • Visual Indicators

11.4.1 How It Works ¶

Each annotation records a text-hash (computed via sxhash-equal) when saved. When the file is next opened and annotations are deserialized, the stored hash is compared against the current text at that position. If they differ, the annotation is flagged as stale.

11.4.2 Visual Indicators ¶

Stale annotations are indicated in several ways:

  • Inline display – A ‘[STALE]’ marker appears in the inline annotation label.
  • Tooltips – The help-echo tooltip is prepended with ‘[STALE]’.
  • Table views – Both single-buffer and project tables include a “Stale” column showing ‘Yes’ (stale), ‘?’ (no stored hash), or blank (fresh).
  • Face – Stale markers use simply-annotate-stale-face (tomato foreground, italic).

Annotations created before stale detection was added will show ‘?’ (unknown) until they are next saved, at which point a hash is recorded.

11.5 Position Tracking ¶

When a file is edited externally (another editor, git operations, merges), lines added or removed above an annotation cause the stored positions to drift. Simply Annotate handles this automatically.

  • How It Works

11.5.1 How It Works ¶

Each annotation stores a text-context field — up to 150 characters of the annotated text. During deserialization, if the content hash does not match at the stored position, the buffer is searched for the stored context. The closest match to the original position is selected and the hash is verified at the new location.

  • If the text is found verbatim at a new position, the overlay is silently relocated there and a ‘[MOVED]’ marker is shown in inline display.
  • If the text is not found (i.e. the annotated text itself was modified), the annotation falls through to stale detection.
  • On the next save, the relocated position is written to the database and the ‘[MOVED]’ indicator clears.

16.1 Storage Format ¶

Annotations are stored in a single Emacs Lisp file (by default ~/.emacs.d/simply-annotations.el). The format is human-readable and can be version controlled if desired.

16.2 Sample Storage Format ¶

With threading support, the storage format includes complete thread structures: 

;;; Simply Annotate Database
;;; This file is auto-generated. Do not edit manually.

(("simply-annotate:/path/to/file.el"
  ((start . 1250) (end . 1300)
   (text . ((id . "thread-123456")
            (created . "2025-06-29T10:30:00")
            (status . "open")
            (priority . "high")
            (tags . ("review" "urgent"))
            (comments . (((id . "c-1719654600-000001")
                          (parent-id . nil)
                          (author . "John Doe")
                          (timestamp . "2025-06-29T10:30:00")
                          (text . "This needs refactoring")
                          (type . "comment"))
                         ((id . "c-1719655500-000002")
                          (parent-id . "c-1719654600-000001")
                          (author . "Jane Smith")
                          (timestamp . "2025-06-29T10:45:00")
                          (text . "I agree, will fix this")
                          (type . "reply"))))))
   (level . defun))
  ((start . 2100) (end . 2150)
   (text . "Add error handling here")
   (level . line)))
 ("simply-annotate:/path/to/other-file.py"
  ((start . 450) (end . 500)
   (text . "Performance bottleneck")
   (level . file))))

16.3 Data Structure Evolution ¶

Simply Annotate supports both legacy string annotations and new thread structures:

  • Legacy format: Simple strings are automatically converted to thread structures when edited
  • Thread format: Complete data structures with metadata and conversation history
  • Level field: Each annotation includes a level field (file, defun, or line)
  • Backward compatibility: Old annotations without a level field default to defun. No migration required
  • Comment IDs: Comments now include id and parent-id fields for nested threading. Legacy comments without these fields are automatically migrated on first access, with replies becoming children of the root comment
  • Migration: Annotations are upgraded to thread format when modified. Use M-x simply-annotate-migrate-threading to migrate the entire database at once

16.4 Data Management ¶

  • The database file is automatically created when first needed
  • Annotations are saved automatically when buffers are saved or killed
  • The file uses a file key system to track annotations per file
  • Empty databases are automatically cleaned up
  • Only saves when there are actual changes to annotations
  • Thread structures are validated when saving to prevent corruption

16.5 File Key System ¶

Each file is identified by a unique key:

  • Uses the full file path for file-visiting buffers
  • Uses the buffer name for non-file buffers
  • Ensures annotations persist across Emacs sessions
  • Supports moving files (annotations follow the buffer name)

16.6 Sexp Editing and Data Integrity ¶

When using sexp editing mode:

  • Data structures are validated before saving
  • Required fields are automatically added if missing
  • Malformed structures are rejected with helpful error messages
  • Backup functionality preserves data before dangerous operations
  • Thread IDs and timestamps are preserved across edits

17.1 Annotations Not Persisting ¶

If your annotations aren’t being saved between sessions:

  • Check that simply-annotate-file is writable
  • Ensure annotation mode is properly enabled
  • Verify that hooks are installed correctly
  • Check file permissions on the storage directory
  • Look for error messages in the *Messages* buffer
  • Verify threading data structure isn’t corrupted

17.2 Display Issues ¶

  • Highlighting Not Visible
  • Fringe Indicators Not Showing

17.2.1 Highlighting Not Visible ¶

  • Customize simply-annotate-highlight-face for your color theme
  • Check for conflicting overlays from other packages
  • Try switching to fringe mode: (setq simply-annotate-display-style 'fringe)
  • Verify the face definition with M-x describe-face

17.2.2 Fringe Indicators Not Showing ¶

  • Ensure your Emacs build supports fringe bitmaps
  • Check that fringe is visible: (set-window-fringes nil 8 8)
  • Try a different fringe indicator: (setq simply-annotate-fringe-indicator 'left-triangle)
  • Verify fringe face colors are visible against your theme

17.3 Smart Action Issues ¶

  • Remember the command is context-sensitive - behavior changes based on region selection and cursor position
  • Use C-u M-s j to force edit mode on existing annotations
  • Check if you’re properly positioned on annotated text (especially in fringe mode)
  • Verify the annotation buffer is responding to C-c C-c and C-c C-k

17.4 Threading Issues ¶

  • Author Prompting Problems
  • Thread Display Corruption
  • Reply Issues

17.4.1 Author Prompting Problems ¶

  • Check simply-annotate-prompt-for-author setting
  • Verify simply-annotate-author-list contains valid names
  • Check that prompting mode matches your workflow needs

17.4.2 Thread Display Corruption ¶

  • Try M-s ' to refresh display style
  • Check for malformed thread data structures
  • Use sexp editing mode to inspect and fix thread structure
  • Verify all required fields are present in thread data

17.4.3 Reply Issues ¶

  • Ensure you’re using C-c C-c to save replies properly
  • Check that M-s r is working on existing annotations
  • Verify author configuration for multi-user workflows
  • Look for timestamp or author field corruption

17.5 Sexp Editing Issues ¶

  • Parsing Errors
  • Data Structure Problems
  • Validation Failures

17.5.1 Parsing Errors ¶

  • Ensure proper Elisp syntax with balanced parentheses
  • Check that all strings are properly quoted
  • Verify required fields (id, comments) are present
  • Use C-c C-k to cancel if unsure about changes

17.5.2 Data Structure Problems ¶

  • Check timestamp format: use ISO format "2025-06-29T10:30:00"
  • Ensure comment type field is "comment" or "reply"
  • Verify status and priority values are from allowed lists
  • Confirm tags field is a list, even if empty: ()

17.5.3 Validation Failures ¶

  • Thread structure must be an association list with required keys
  • comments field must be a list of comment objects
  • Each comment must have author, timestamp, text, and type
  • Use the helpful comments in sexp editing mode as a guide

17.6 Performance with Large Files ¶

Simply Annotate is designed to be lightweight:

  • Annotations use lightweight overlays
  • No performance impact on file operations
  • Fringe mode generally has better performance than highlight mode
  • Sexp editing may be slower with very large annotation structures
  • Consider splitting very large annotation sets across multiple files

17.7 Integration Issues ¶

  • Annotations work with most major modes
  • Compatible with lsp-mode, flycheck, and other packages
  • May need customization for exotic display modes
  • Check for conflicting key bindings with C-h k
  • Fringe indicators work well with line number modes
  • Sexp editing works in any buffer that supports Emacs Lisp editing

17.8 Buffer-Local Variable Issues ¶

If annotations disappear when opening new files:

  • Ensure buffer-local variables are properly declared
  • Check that simply-annotate-overlays is buffer-local
  • Verify mode activation is working correctly per buffer
  • Use debug functions to check overlay state

17.9 Common Error Messages ¶

"No annotation at point"

You’re trying to edit or remove an annotation, but the cursor isn’t positioned on annotated text. In fringe mode, make sure you’re on the correct line.

"Annotation cancelled/removed"

You saved an empty annotation, which automatically removes it.

"No annotations in buffer"

The current buffer has no annotations to navigate or list.

"No annotations database found"

No annotation file exists yet. Create your first annotation to initialize the database.

"Invalid annotation format: must be a thread alist"

Sexp editing validation failed. Check the data structure format and required fields.

"Error parsing sexp"

The Elisp syntax in sexp editing mode is invalid. Check for balanced parentheses and proper quoting.

Database file errors

Check permissions and disk space for the storage file location.

17.10 Getting Help ¶

  • Use C-h k followed by a key binding to see what command it runs
  • Check the *Messages* buffer for error messages
  • Use M-x describe-mode in an annotation buffer for mode-specific help
  • The header-line always shows available commands and current status
  • Use sexp editing mode to inspect annotation data structures directly
  • Check threading information in the annotation list (M-s l) for debugging
