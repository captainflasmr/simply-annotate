** 1 Introduction [[#Introduction-1][¶]]
:PROPERTIES:
:CUSTOM_ID: Introduction-1
:CLASS: chapter
:END:
Simply Annotate is a lightweight annotation system for Emacs that allows
you to add persistent notes to any text file, or specialized non-file
buffers like =Info-mode= manuals, without modifying the original
content. It's perfect for code review, research, note-taking, and
documentation and includes threading and collaboration features.

- [[#Key-Features][Key Features]]

<<Key-Features>>
*** 1.1 Key Features [[#Key-Features][¶]]
:PROPERTIES:
:CUSTOM_ID: key-features
:CLASS: section
:END:
- *Non-intrusive*: Annotations don't modify your original files
- *Persistent*: Notes survive between Emacs sessions
- *Threading*: Add replies to annotations for conversations
- *Support for Non-File Buffers*: Annotate Info manuals or other
  specialized buffers natively
- *Status tracking*: Manage annotation lifecycle (open → resolved)
- *Multi-author support*: Collaborate with team members
- *Priority levels*: Organize by importance (low, normal, high,
  critical)
- *Tag system*: Categorize with hashtags (=#review=, =#bug=,
  =#question=)
- *Raw sexp editing*: Direct access to annotation data structures for
  power users
- *Annotation levels*: Three-tier annotation system (file, defun, line)
  for progressive code exploration
- *Multiple display styles*: Choose between highlighting, tint, fringe,
  fringe bracket, both, or subtle
- *Smart action*: Context-aware command that adapts to the situation
- *Rich editing*: Full-featured annotation buffer for longer notes
- *Navigation*: Jump between annotations with simple keybindings
- *Overview*: List all annotations in an org-mode buffer with folding
- *Cross-file*: Browse all annotations across files in a global org-mode
  overview
- *Org-mode export*: Convert annotation threads to structured TODO items
- *Header-line integration*: Shows annotation count and available
  commands
- *Dired integration*: Fringe indicators in dired for files with
  annotations
- *Kanban board*: Visual board view of project annotations grouped by
  status columns

*** 2.1 Using MELPA (Recommended) [[#Using-MELPA-_0028Recommended_0029][¶]]
:PROPERTIES:
:CUSTOM_ID: using-melpa-recommended
:CLASS: section
:END:
The easiest way to install Simply Annotate is through MELPA:

#+begin_src lisp-preformatted
(use-package simply-annotate
  :ensure t
  :demand t
  :hook (find-file-hook . simply-annotate-mode)
  :config
  (global-set-key (kbd "M-s") simply-annotate-command-map))
#+end_src

<<Keymap-Configuration>>
*** 2.2 Keymap Configuration [[#Keymap-Configuration][¶]]
:PROPERTIES:
:CUSTOM_ID: keymap-configuration
:CLASS: section
:END:
Simply Annotate provides two keymaps:

- =simply-annotate-mode-map= :: The minor mode keymap, intentionally
  minimal. Only binds M-n (next annotation) and M-p (previous
  annotation) so it won't conflict with other packages.

- =simply-annotate-command-map= :: The main command keymap with short
  single-key bindings (j, r, n, e, etc.). You bind this to a prefix key
  of your choice.

The recommended prefix is M-s, which gives you M-s j (smart action), M-s
r (reply), M-s l (list), M-s C (clear), etc. All keybindings in this
manual assume the M-s prefix.

*Important:* Use =:demand t= with =global-set-key= in =:config= for the
M-s prefix. =:demand t= loads the package immediately so the prefix is
available everywhere in Emacs from startup.

Do not use =:bind-keymap= for M-s. =:bind-keymap= defers loading by
creating an autoload proxy --- a temporary non-prefix command that loads
the package on first keypress. This works well for free keys like C-c a,
but M-s is already Emacs's =search-map= prefix. The proxy replaces
=search-map= with a non-prefix command, so any code that subsequently
tries to bind under M-s will error. Using =global-set-key= in =:config=
avoids this by binding directly to the real keymap after the package has
loaded.

If M-s conflicts with other packages (e.g. Consult), use =:bind-keymap=
with a free prefix like C-c a:

#+begin_src lisp-preformatted
;; C-c a prefix --- :bind-keymap works well here because
;; C-c a is a free key with no existing prefix map, and
;; it defers loading until first use.
(use-package simply-annotate
  :bind-keymap ("C-c a" . simply-annotate-command-map)
  :hook (find-file-hook . simply-annotate-mode))
#+end_src

<<Manual-Installation>>
*** 2.3 Manual Installation [[#Manual-Installation][¶]]
:PROPERTIES:
:CUSTOM_ID: manual-installation

:END:
If you prefer to install manually:

1. Download =simply-annotate.el=
2. Add to your load path and bind the command map:

   #+begin_src lisp-preformatted

** 3 Quick Start [[#Quick-Start-1][¶]]
:PROPERTIES:
:CUSTOM_ID: Quick-Start-1
:CLASS: chapter
:END:
Here's how to get started with Simply Annotate:

1. Open any file in Emacs
2. Enable annotation mode: M-x simply-annotate-mode
3. Select some text and press M-s j to create your first annotation
4. Type your annotation text and press C-c C-c to save
5. Navigate with M-n (next) and M-p (previous)
6. Press M-s ' to cycle through different display styles
7. Press M-s [ and M-s ] to cycle through annotation levels
8. Press M-s r to add a reply to any annotation
9. Press M-s s to set status (open, resolved, etc.)
10. Press M-s e to edit annotation data structure directly
11. Press M-s o to export annotations to org-mode

<<Smart-Action-Command>>
**** 4.2.1 Smart Action Command [[#Smart-Action-Command][¶]]
:PROPERTIES:
:CUSTOM_ID: smart-action-command
:CLASS: subsection
:END:
The M-s j command (=simply-annotate-smart-action=) is context-aware and
behaves differently depending on the situation:

- *With region selected* :: Creates new annotation or edits existing one
  at the selected text

- *On annotated text (no region)* :: Toggles annotation buffer
  visibility to show/hide the annotation

- *With prefix argument (C-u M-s j)* :: Forces edit mode on existing
  annotation

- *Elsewhere (no region, no annotation)* :: Creates annotation for the
  current line

<<Traditional-Method>>
**** 4.2.2 Traditional Method [[#Traditional-Method][¶]]
:PROPERTIES:
:CUSTOM_ID: traditional-method
:CLASS: subsection
:END:
For more explicit control:

1. Select the region you want to annotate
2. Press M-s j
3. Enter your annotation text in the dedicated buffer
4. Save with C-c C-c

<<Multi_002dline-Annotations>>
**** 4.2.3 Multi-line Annotations [[#Multi_002dline-Annotations][¶]]
:PROPERTIES:
:CUSTOM_ID: multi-line-annotations
:CLASS: subsection
:END:
The annotation buffer supports full multi-line editing:

1. Select the region you want to annotate
2. Press M-s j
3. Use the full annotation buffer for detailed notes
4. Format your text as needed (multiple paragraphs, bullet points, etc.)
5. Save with C-c C-c

<<Viewing-Annotations>>
*** 4.3 Viewing Annotations [[#Viewing-Annotations][¶]]
:PROPERTIES:
:CUSTOM_ID: viewing-annotations
:CLASS: section
:END:
When =simply-annotate-mode= is active:

- Annotated text is displayed according to your chosen display style
- The header line shows annotation count and available commands
- Thread status and comment count displayed in header (e.g.,
  =[OPEN/HIGH:3]=)
- Moving to annotated text shows annotation details in the header
- Press M-s j on annotated text to view/edit in detail

<<Editing-Annotations>>
*** 4.4 Editing Annotations [[#Editing-Annotations][¶]]
:PROPERTIES:
:CUSTOM_ID: editing-annotations
:CLASS: section
:END:
To edit an existing annotation:

1. Place cursor on annotated text
2. Press M-s j to open the annotation buffer
3. Make your changes directly (the buffer starts in edit mode)
4. Save with C-c C-c or cancel with C-c C-k

<<Deleting-Annotations>>
*** 4.5 Deleting Annotations [[#Deleting-Annotations][¶]]
:PROPERTIES:
:CUSTOM_ID: deleting-annotations
:CLASS: section
:END:
To remove an annotation:

1. Place cursor on annotated text
2. Press M-s - to remove the annotation

Alternatively, to remove all annotations from the current buffer at
once:

1. Press M-s C
2. Confirm the deletion at the prompt (y or n)

Alternatively, you can edit the annotation and delete all text, then
save with C-c C-c to remove it.

<<Sexp-Structure>>
**** 5.1.2 Sexp Structure [[#Sexp-Structure][¶]]
:PROPERTIES:
:CUSTOM_ID: sexp-structure
:CLASS: subsection
:END:
Annotations are stored as Elisp association lists with the following
structure:

#+begin_src lisp-preformatted
((id . "thread-123456")
 (created . "2025-06-29T10:30:00")
 (status . "open")
 (priority . "high")
 (tags . ("bug" "critical"))
 (comments . (((id . "c-1719654600-000001")
               (parent-id . nil)
               (author . "John Doe")
               (timestamp . "2025-06-29T10:30:00")
               (text . "Found a critical bug here")
               (type . "comment"))
              ((id . "c-1719657300-000002")
               (parent-id . "c-1719654600-000001")
               (author . "Jane Smith")
               (timestamp . "2025-06-29T11:15:00")
               (text . "I can reproduce this issue")
               (type . "reply")))))
#+end_src

<<Required-Fields>>
**** 5.1.3 Required Fields [[#Required-Fields][¶]]
:PROPERTIES:
:CUSTOM_ID: required-fields
:CLASS: subsection
:END:
- <<index-id>>=id= [[#index-id][¶]] :: Unique identifier for the thread
  (string)

- <<index-created>>=created= [[#index-created][¶]] :: ISO timestamp when
  thread was created

- <<index-status-1>>=status= [[#index-status-1][¶]] :: Current status
  (="open"=, ="in-progress"=, ="resolved"=, ="closed"=)

- <<index-priority-1>>=priority= [[#index-priority-1][¶]] :: Priority
  level (="low"=, ="normal"=, ="high"=, ="critical"=)

- <<index-tags-1>>=tags= [[#index-tags-1][¶]] :: List of tag strings
  (can be empty list)

- <<index-comments>>=comments= [[#index-comments][¶]] :: List of
  comment/reply objects

<<Comment-Structure>>
**** 5.1.4 Comment Structure [[#Comment-Structure][¶]]
:PROPERTIES:
:CUSTOM_ID: comment-structure
:CLASS: subsection
:END:
Each comment in the =comments= list must have:

- <<index-id-1>>=id= [[#index-id-1][¶]] :: Unique identifier for the
  comment (string, e.g. ="c-1719654600-000001"=). Auto-assigned for
  legacy comments on first access.

- <<index-parent_002did>>=parent-id= [[#index-parent_002did][¶]] :: The
  =id= of the parent comment this replies to, or =nil= for root
  comments. This field encodes the nested reply hierarchy.

- <<index-author>>=author= [[#index-author][¶]] :: Author name (string)

- <<index-timestamp>>=timestamp= [[#index-timestamp][¶]] :: ISO
  timestamp when comment was created

- <<index-text>>=text= [[#index-text][¶]] :: The actual comment text
  content

- <<index-type>>=type= [[#index-type][¶]] :: Either ="comment"=
  (original) or ="reply"=

<<Adding-Replies>>
**** 6.1.1 Adding Replies [[#Adding-Replies][¶]]
:PROPERTIES:
:CUSTOM_ID: adding-replies
:CLASS: subsection
:END:
To add a reply to any annotation:

1. Place cursor on annotated text
2. Press M-s r
3. If the thread has multiple comments, select which comment to reply to
   from the hierarchical =completing-read= menu
4. Enter your reply text
5. The reply is nested under the selected comment

Replies support arbitrary nesting depth. When the thread has only one
comment, the reply targets it directly with no extra prompt.

<<Thread-Display>>
**** 6.1.2 Thread Display [[#Thread-Display][¶]]
:PROPERTIES:
:CUSTOM_ID: thread-display
:CLASS: subsection
:END:
Threaded annotations are displayed with hierarchical indentation:

#+begin_src example-preformatted
┌─ ✎ [OPEN/HIGH] review, urgent ────────────
│ John Doe (06/19 09:49):
│  This needs review before release
│   ↳ Jane Smith (06/19 10:15):
│     I'll check this today
│     ↳ Bob Wilson (06/19 10:17):
│       Ok!
│       ↳ Jane Smith (06/19 18:17):
│         Done!
│   ↳ Bob Wilson (06/19 11:30):
│     Looks good to me, approved
└──────────────────────────────────────────┘
#+end_src

Each depth level adds 2 spaces of indentation. The same hierarchical
rendering is used in inline overlays, the listing buffer (nested org
headings), and org export.

<<Database-Migration>>
**** 6.1.3 Database Migration [[#Database-Migration][¶]]
:PROPERTIES:
:CUSTOM_ID: database-migration
:CLASS: subsection
:END:
Existing annotations from before version 0.9.0 use a flat comment list
without =id= or =parent-id= fields. These are automatically migrated on
first access: replies (comments with =type = "reply"=) become children
of the root comment.

To migrate the entire database at once, run:

#+begin_src example-preformatted
M-x simply-annotate-migrate-threading
#+end_src

This is safe to run multiple times and only modifies threads that need
updating.

**** 6.2.5 Searching by Tag [[#Searching-by-Tag][¶]]
:PROPERTIES:
:CUSTOM_ID: searching-by-tag
:CLASS: subsection
:END:

tag. The command uses a two-stage workflow:

1. Select a tag from all tags in the current scope via =completing-read=
2. Select a matching annotation from the filtered results
3. Jump to the annotation's source file and position

The scope is controlled by prefix arguments:

- M-s F :: Project files only (or all files when not in a project). When
  called from a =dired= buffer under a project subdirectory, candidates
  are automatically narrowed to that subdirectory.

- C-u M-s F :: Whole project, ignoring any dired auto-narrow.

- C-u C-u M-s F :: Every annotated file in the database.

Simply Annotate supports five display styles that can be used
individually or combined as a list. Styles are processed in order, so
later face-based styles override earlier ones while fringe styles are
independent.

<<Available-Styles>>
*** 7.1 Available Styles [[#Available-Styles][¶]]
:PROPERTIES:
:CUSTOM_ID: available-styles
:CLASS: section
:END:
- =highlight= :: Traditional background highlighting of annotated text
  (good for emphasis)

- =tint= :: Subtle background colour derived from the current theme,
  lightened by =simply-annotate-tint-amount= (default 5%). Adapts
  automatically to any theme.

- =fringe= :: Shows small indicators in the left fringe (minimal visual
  impact)

- =fringe-bracket= :: Draws a vertical bracket in the left fringe
  spanning the full annotated region. Uses the configured fringe
  indicator at the top with a continuous vertical bar and a horizontal
  cap at the bottom. Customisable via
  =simply-annotate-fringe-bracket-face=.

- =bar= :: Draws a solid heavy vertical bar (=┃ =) in the left margin of
  every annotated line.

- =bracket= :: Outlines annotated regions with heavy box-drawing corners
  (=┏ =, =┃ =, =┗ =) for multi-line annotations, or a single =┠ =
  indicator for single-line annotations. This is now the default display
  style.

- =subtle= :: Uses overline and underline to bracket annotated regions
  (unobtrusive)

<<Combining-Styles>>

#+begin_src lisp-preformatted

(setq simply-annotate-display-style '(tint fringe-bracket))
#+end_src

Additionally, inline display can be toggled on top of any style (see
[[#Display-Styles][Display Styles]]).

<<Inline-Display>>
*** 7.6 Inline Display [[#Inline-Display][¶]]
:PROPERTIES:
:CUSTOM_ID: inline-display
:CLASS: section
:END:
Inline display is an overlay that can be toggled on top of any display
style with M-s / (=simply-annotate-toggle-inline=). When active, it
renders annotation text as a box-drawn block relative to the annotated
region.

The position is controlled by =simply-annotate-inline-position=:

- =after= :: (Default) The block appears after the annotated region:

  #+begin_src example-preformatted
  (defun calculate-total (items)
    ┌─ ✎ [OPEN/NORMAL] ────────
    │ Sums all item prices.
    │ Returns zero for empty list.
    └────────────────────────────
    (apply #'+ (mapcar #'price items)))
  #+end_src

- =above= :: The block appears above the annotated region:

  #+begin_src example-preformatted
    ┌─ ✎ [OPEN/NORMAL] ────────
    │ Sums all item prices.
    │ Returns zero for empty list.
    └────────────────────────────
  (defun calculate-total (items)
    (apply #'+ (mapcar #'price items)))
  #+end_src

Both inline blocks and the underlying display style (highlight, fringe,
both, or subtle) are filtered to the active annotation level --- cycling
levels with M-s [ updates which annotations are visible.

Long annotations are truncated after =simply-annotate-inline-max-lines=
lines (default 20). Thread metadata (status, priority, comment count)
appears in the header.

- [[#Inline-Pointer][Inline Pointer]]
- [[#Inline-Customisation][Inline Customisation]]

<<Inline-Pointer>>
**** 7.6.1 Inline Pointer [[#Inline-Pointer][¶]]
:PROPERTIES:
:CUSTOM_ID: inline-pointer
:CLASS: subsection
:END:
A pointer connects each inline box to the annotated text, making the
association visually clear. The pointer string is indented to the
annotation start column. Strings can be multiline (use =\n=). Set to
=nil= or an empty string to disable.

- =simply-annotate-inline-pointer-after= :: Pointer shown between
  annotated text and the box below. Default: ="▴"=

- =simply-annotate-inline-pointer-above= :: Pointer shown between the
  box above and annotated text. Default: ="▾"=

Example pointer styles (see the Commentary section for a full list):

#+begin_src example-preformatted
;; Simple arrows (default)
(setq simply-annotate-inline-pointer-after "▴")
(setq simply-annotate-inline-pointer-above "▾")

;; Heavy L-bracket
(setq simply-annotate-inline-pointer-after "┃\n┗━▶")
(setq simply-annotate-inline-pointer-above "┏━▶\n┃")

;; Disabled
(setq simply-annotate-inline-pointer-after nil)
(setq simply-annotate-inline-pointer-above nil)
#+end_src

<<Inline-Customisation>>
**** 7.6.2 Inline Customisation [[#Inline-Customisation][¶]]
:PROPERTIES:
:CUSTOM_ID: inline-customisation
:CLASS: subsection

Customise the appearance with:

Simply Annotate provides a three-tier annotation level system for

<<Level-Overview>>
*** 8.1 Level Overview [[#Level-Overview][¶]]
:PROPERTIES:
:CUSTOM_ID: level-overview
:CLASS: section
:END:
- =file= :: Whole-file descriptions. Use this level for high-level
  summaries of what a file does, its purpose in the project, or overall
  architecture notes.

- =defun= :: Function or element descriptions. The default level,
  suitable for documenting individual functions, classes, or major code
  blocks.

- =line= :: Individual syntactical elements. Use this level for
  fine-grained notes about specific expressions, variables, or
  implementation details.

<<How-Levels-Work>>
*** 8.2 How Levels Work [[#How-Levels-Work][¶]]
:PROPERTIES:
:CUSTOM_ID: how-levels-work
:CLASS: section
:END:
Only annotations matching the active level are styled. Annotations at
other levels have their visual markers (highlights, fringe indicators,
inline blocks) removed, but the underlying text remains visible:

- *Display*: Only overlays matching the active level show highlights,
  fringe indicators, and inline blocks.
- *Navigation*: M-n and M-p visit only annotations at the active level.
- *Editing*: M-s j on annotated text only opens annotations matching the
  active level.
- *Header display*: The annotation count and position indicator reflect
  only the active level.

<<Cycling-Levels>>
*** 8.3 Cycling Levels [[#Cycling-Levels][¶]]
:PROPERTIES:
:CUSTOM_ID: cycling-levels
:CLASS: section
:END:
Use M-s ] to cycle forward and M-s [ to cycle backward through levels:

#+begin_src example-preformatted
M-s ]: file ⇒ defun ⇒ line ⇒ file ⇒ ...
M-s [: line ⇒ defun ⇒ file ⇒ line ⇒ ...
#+end_src

The header-line shows annotation counts for every level, with the active
level in bold (e.g. =FILE:2 | DEFUN:5 | LINE:3=).

<<Direct-Level-Selection>>
*** 8.4 Direct Level Selection [[#Direct-Level-Selection][¶]]
:PROPERTIES:
:CUSTOM_ID: direct-level-selection
:CLASS: section
:END:
Use =M-x simply-annotate-set-level= to jump directly to a specific level
via =completing-read=.

:CUSTOM_ID: creating-annotations-at-a-level
:CLASS: section
:END:
New annotations inherit the currently active level. For example, if you
cycle to =file= level and then create an annotation, it will be tagged

- M-n :: =simply-annotate-next= - Jump to next annotation

the active annotation level---only annotations matching the current

highlighted with a pulse effect.

<<Listing-All-Annotations>>
*** 10.1 Listing All Annotations [[#Listing-All-Annotations][¶]]
:PROPERTIES:
:CUSTOM_ID: listing-all-annotations
:CLASS: section
:END:
To see an overview of all annotations in the current buffer:

#+begin_src example-preformatted
M-s l
#+end_src

The listing uses org-mode for full folding support. The structure is:

#+begin_src example-preformatted
,#+TITLE: Annotations for myfile.el
,#+STARTUP: show2levels

,* FILE (1)
,** [OPEN/NORMAL] Line 1 (1 comment)
,*** Region
~;; file header comment~
,*** Comment - james dyer (06/19 09:49)
Overall file description

,* DEFUN (2)
,** [OPEN/HIGH] Line 15 #review (2 comments)
,*** Region
~(defun calculate-total ()~
,*** Comment - james dyer (06/19 09:49)
This needs review before release
,*** Reply - jane smith (06/19 10:15)
I'll check this today
#+end_src

Use TAB and S-TAB for org folding. Within the listing buffer:

- n -- move to next annotation and jump to source
- p -- move to previous annotation and jump to source
- RET -- jump to source location of annotation at point
- q -- quit the listing buffer

The listing automatically refreshes when annotations are modified
(saved, edited, replied to, removed, or have properties changed).

<<Global-Annotation-Overview>>
*** 10.2 Global Annotation Overview [[#Global-Annotation-Overview][¶]]
:PROPERTIES:
:CUSTOM_ID: global-annotation-overview
:CLASS: section
:END:
To see all annotations across all files in a single org-mode buffer,
bind =simply-annotate-show-all= globally:

#+begin_src lisp-preformatted
(global-set-key (kbd "C-c 0") #'simply-annotate-show-all)
#+end_src

This produces a =*All Annotations*= buffer with files as top-level
headings:

#+begin_src example-preformatted
,#+TITLE: All Annotations
,#+STARTUP: content

,* ~/project/myfile.el (3)
,** FILE (1)
,*** [OPEN/NORMAL] Line 1 (1 comment)
,** DEFUN (2)
,*** [OPEN/HIGH] Line 15 #review (2 comments)

,* ~/project/other.el (5)
,** DEFUN (3)
...
#+end_src

The same n/p/RET/q navigation works here. Jumping to an annotation opens
the source file in another window.

<<Quick-File-Jump>>
*** 10.3 Quick File Jump [[#Quick-File-Jump][¶]]
:PROPERTIES:
:CUSTOM_ID: quick-file-jump
:CLASS: section
:END:
For quick navigation without the full overview, bind
=simply-annotate-jump-to-file= globally:

#+begin_src lisp-preformatted
(global-set-key (kbd "C-c 9") #'simply-annotate-jump-to-file)
#+end_src

This uses =completing-read= to show all annotated files with counts,
opens the selected file, and enables =simply-annotate-mode=.

<<Dired-Integration>>
*** 10.4 Dired Integration [[#Dired-Integration][¶]]
:PROPERTIES:
:CUSTOM_ID: dired-integration
:CLASS: section
:END:
Enable =simply-annotate-dired-mode= in dired buffers to show fringe
indicators next to files that have annotations in the database:

#+begin_src lisp-preformatted
;; Enable for all dired buffers
(add-hook 'dired-mode-hook #'simply-annotate-dired-mode)
#+end_src

The fringe indicator uses the same symbol and face as annotation markers
in source buffers (=simply-annotate-fringe-indicator= and
=simply-annotate-fringe-face=). Indicators refresh automatically when
the dired buffer is re-read (e.g. after pressing g).

<<Annotation-Buffer-Features>>
*** 10.5 Annotation Buffer Features [[#Annotation-Buffer-Features][¶]]
:PROPERTIES:
:CUSTOM_ID: annotation-buffer-features
:CLASS: section
:END:
The annotation buffer provides a rich editing environment with threading
support:

- Shows helpful instructions with threading commands in the header
- Displays thread information, status, and priority
- Shows complete conversation thread with replies
- Access to threading commands (M-s r, M-s s, M-s p, M-s t)
- Access to advanced sexp editing with M-s e
- Supports multi-line annotations with full editing capabilities
- Starts in edit mode for immediate typing
- Provides save (C-c C-c) and cancel (C-c C-k) operations
- Automatically handles empty annotations (removes them)

<<Header_002dline-Integration>>
*** 10.6 Header-line Integration [[#Header_002dline-Integration][¶]]
:PROPERTIES:
:CUSTOM_ID: header-line-integration
:CLASS: section
:END:
The header-line provides real-time information with threading support:

- Active annotation level indicator (=[FILE]=, =[DEFUN]=, or =[LINE]=)
- Current annotation position scoped to the active level (e.g., "2/5")
- Thread status and priority (e.g., =[OPEN/HIGH:3]=)
- Total annotation count for the active level
- Enhanced keybinding help including threading and level commands
- Context-sensitive help text
- Automatic updates when annotations, threads, or levels change

** 11 Project-Aware Annotations [[#Project_002dAware-Annotations-1][¶]]
:PROPERTIES:
:CUSTOM_ID: Project_002dAware-Annotations-1
:CLASS: chapter
:END:
By default, all annotations are stored in a single global database file.
Version 1.0.0 adds project-aware storage and project-scoped views, using
Emacs built-in =project.el=.

- [[#Database-Strategy][Database Strategy]]
- [[#Migrating-to-a-Project-Database][Migrating to a Project Database]]
- [[#Project_002dScoped-Views][Project-Scoped Views]]
- [[#Stale-Annotation-Detection][Stale Annotation Detection]]
- [[#Position-Tracking][Position Tracking]]

<<Database-Strategy>>
*** 11.1 Database Strategy [[#Database-Strategy][¶]]
:PROPERTIES:
:CUSTOM_ID: database-strategy
:CLASS: section
:END:
The =simply-annotate-database-strategy= variable controls where
annotations are stored:

- <<index-global>>=global= [[#index-global][¶]] :: Single database at
  =simply-annotate-file= (default, existing behaviour).

- <<index-project>>=project= [[#index-project][¶]] :: Per-project
  database at the project root, falling back to the global database for
  files not inside a recognised project. File keys are stored as
  project-relative paths, making the database portable and committable.

- <<index-both>>=both= [[#index-both][¶]] :: Write to the project
  database when inside a project; read from both project and global
  databases, with project entries taking precedence on conflicts.
  Annotations from the global database migrate to the project database
  when their files are next saved.

The =simply-annotate-project-file= variable (default
=.simply-annotations.el=) sets the filename placed at the project root.

- [[#Setup][Setup]]

<<Setup>>
**** 11.1.1 Setup [[#Setup][¶]]
:PROPERTIES:
:CUSTOM_ID: setup
:CLASS: subsection
:END:

#+begin_src example-preformatted
;; Store annotations at the project root
(setq simply-annotate-database-strategy 'project)

;; Or merge project and global databases
(setq simply-annotate-database-strategy 'both)
#+end_src

To share annotations with your team, commit =.simply-annotations.el=. To
keep annotations private, add it to =.gitignore=.

<<Migrating-to-a-Project-Database>>
*** 11.2 Migrating to a Project Database [[#Migrating-to-a-Project-Database][¶]]
:PROPERTIES:
:CUSTOM_ID: migrating-to-a-project-database
:CLASS: section
:END:
When switching from =global= to =project= strategy, existing annotations
live in the global database with absolute file paths. The migration
command copies matching annotations into a project-local database with
relative keys:

#+begin_src example-preformatted
M-x simply-annotate-migrate-to-project
#+end_src

This command:

1. Reads the global database
2. Finds entries whose file paths fall under the current project root
3. Writes them to the project database with relative keys
4. Removes migrated entries from the global database

You must have a file open inside the target project before running this
command, so that =project-current= can detect the project root.

<<Project_002dScoped-Views>>
*** 11.3 Project-Scoped Views [[#Project_002dScoped-Views][¶]]
:PROPERTIES:
:CUSTOM_ID: project-scoped-views
:CLASS: section
:END:
=simply-annotate-show-project= (P in the command map) shows an org-mode
listing of all annotations in the current project, grouped by directory
and file. Like =simply-annotate-show-all= but filtered to the current
project.

=simply-annotate-list-project-table= (C-t in the command map) opens a
sortable table spanning all project files, with columns for File, Level,
Stale, Line, Status, Priority, Comments, Tags, Author, and Comment.

=simply-annotate-jump-to-file= (f in the command map) defaults to
project files when using =project= or =both= strategy. Use a prefix
argument (C-u f) to show all files across all databases.

<<Stale-Annotation-Detection>>
*** 11.4 Stale Annotation Detection [[#Stale-Annotation-Detection][¶]]
:PROPERTIES:
:CUSTOM_ID: stale-annotation-detection
:CLASS: section
:END:
When the text under an annotation changes (refactoring, edits, or
merges), the annotation may no longer apply to the correct content.
Simply Annotate detects this by storing a content hash of the annotated
text region.

- [[#How-It-Works][How It Works]]
- [[#Visual-Indicators][Visual Indicators]]

<<How-It-Works>>
**** 11.4.1 How It Works [[#How-It-Works][¶]]
:PROPERTIES:
:CUSTOM_ID: how-it-works
:CLASS: subsection
:END:
Each annotation records a =text-hash= (computed via =sxhash-equal=) when
saved. When the file is next opened and annotations are deserialized,
the stored hash is compared against the current text at that position.
If they differ, the annotation is flagged as stale.

<<Visual-Indicators>>
**** 11.4.2 Visual Indicators [[#Visual-Indicators][¶]]
:PROPERTIES:
:CUSTOM_ID: visual-indicators
:CLASS: subsection
:END:
Stale annotations are indicated in several ways:

- *Inline display* -- A ‘=[STALE]=' marker appears in the inline
  annotation label.
- *Tooltips* -- The help-echo tooltip is prepended with ‘=[STALE]='.
- *Table views* -- Both single-buffer and project tables include a
  “Stale” column showing ‘=Yes=' (stale), ‘=?=' (no stored hash), or
  blank (fresh).
- *Face* -- Stale markers use =simply-annotate-stale-face= (tomato
  foreground, italic).

Annotations created before stale detection was added will show ‘=?='
(unknown) until they are next saved, at which point a hash is recorded.

<<Position-Tracking>>
*** 11.5 Position Tracking [[#Position-Tracking][¶]]
:PROPERTIES:
:CUSTOM_ID: position-tracking
:CLASS: section
:END:
When a file is edited externally (another editor, git operations,
merges), lines added or removed above an annotation cause the stored
positions to drift. Simply Annotate handles this automatically.

- [[#How-It-Works-1][How It Works]]

<<How-It-Works-1>>
**** 11.5.1 How It Works [[#How-It-Works-1][¶]]
:PROPERTIES:
:CUSTOM_ID: how-it-works-1
:CLASS: subsection
:END:
Each annotation stores a =text-context= field --- up to 150 characters
of the annotated text. During deserialization, if the content hash does
not match at the stored position, the buffer is searched for the stored
context. The closest match to the original position is selected and the
hash is verified at the new location.

- If the text is found verbatim at a new position, the overlay is
  silently relocated there and a ‘=[MOVED]=' marker is shown in inline
  display.
- If the text is not found (i.e. the annotated text itself was
  modified), the annotation falls through to stale detection.
- On the next save, the relocated position is written to the database
  and the ‘=[MOVED]=' indicator clears.

** 12 Kanban Board [[#Kanban-Board-1][¶]]
:PROPERTIES:
:CUSTOM_ID: Kanban-Board-1
:CLASS: chapter
:END:
The kanban board (M-s K or =simply-annotate-kanban=) displays project
annotations as cards arranged in columns by status. It requires a
project detectable by =project.el=.

- [[#Board-Layout][Board Layout]]
- [[#Navigation-2][Navigation]]
- [[#Actions][Actions]]
- [[#Filtering][Filtering]]
- [[#Expand-and-Follow][Expand and Follow]]

<<Board-Layout>>
*** 12.1 Board Layout [[#Board-Layout][¶]]
:PROPERTIES:
:CUSTOM_ID: board-layout
:CLASS: section
:END:
The board renders four columns corresponding to the thread statuses
(=open=, =in-progress=, =resolved=, =closed=). Each card shows:

- Priority indicator with colour coding (critical = red, high = orange,
  etc.)
- First line of the annotation text as a summary
- Source file and line number

#+begin_src example-preformatted
 OPEN (3)            IN-PROGRESS (1)     RESOLVED (0)        CLOSED (0)
────────────────────────────────────────────────────────────────────────
┌──────────────────┐ ┌──────────────────┐
│ [normal] Review.. │ │ [high] Fix bug.. │
│ src/main.el:42   │ │ src/util.el:15   │
└──────────────────┘ └──────────────────┘
┌──────────────────┐
│ [low] Add docs.. │
│ README.org:10    │
└──────────────────┘
#+end_src

<<Navigation-2>>
*** 12.2 Navigation [[#Navigation-2][¶]]
:PROPERTIES:
:CUSTOM_ID: navigation
:CLASS: section
:END:
Navigation follows an org-table-style grid:

- n :: Move to the next card in the same column (down)

- p :: Move to the previous card in the same column (up)

- TAB :: Move to the same row in the next column (right). If the target
  column is shorter, clamps to its last card.

- S-TAB :: Move to the same row in the previous column (left), with the
  same clamping behaviour.

The card under point is highlighted with bold face, providing visual
feedback for which card is active.

<<Actions>>
*** 12.3 Actions [[#Actions][¶]]
:PROPERTIES:
:CUSTOM_ID: actions
:CLASS: section
:END:
- RET :: Jump to the annotation's source file and position.

- s :: Set the card's status via =completing-read=.

- } :: Move the card one status column to the right (e.g. open ⇒
  in-progress).

- { :: Move the card one status column to the left (e.g. in-progress ⇒
  open).

- g :: Refresh the board in place, preserving cursor position.

- q :: Quit the kanban buffer.

Plain string annotations (legacy format) are automatically converted to
thread format when their status is changed from the board.

<<Filtering>>
*** 12.4 Filtering [[#Filtering][¶]]
:PROPERTIES:
:CUSTOM_ID: filtering
:CLASS: section
:END:
The kanban board supports three filters that can be combined (AND
logic). Active filters are shown in the header line with green
highlighting.

- a :: Filter by author. Select from authors present in the current
  board, or choose ‘=[All]=' to clear.

- l :: Filter by annotation level (=file=, =defun=, =line=). Select a
  level or ‘=[All]=' to clear.

- t :: Filter by tag. Select from tags present in the current board, or
  choose ‘=[All]=' to clear.

- d :: Clear the directory filter (widening a dired-narrowed board back
  to the whole project).

Filters stack: for example, pressing a to select an author and then t to
select a tag shows only cards by that author with that tag.

<<Expand-and-Follow>>
*** 12.5 Expand and Follow [[#Expand-and-Follow][¶]]
:PROPERTIES:
:CUSTOM_ID: expand-and-follow
:CLASS: section
:END:
- e :: Toggle expanded view for the card at point. Expanded cards show
  full annotation text, tags, and reply count.

- E :: Toggle expand/collapse for all cards at once.

- F :: Toggle follow mode. When on, navigating between cards
  automatically shows the annotation source in another window.

<<Exporting-Annotations>>
*** 13.1 Exporting Annotations [[#Exporting-Annotations][¶]]
:PROPERTIES:
:CUSTOM_ID: exporting-annotations
:CLASS: section
:END:
To export current buffer annotations to org-mode:

#+begin_src example-preformatted
M-s o
#+end_src

This prompts for a filename and creates a structured org-mode file with:

- Each annotation thread as a TODO item
- Complete metadata preserved in properties
- Replies as nested sub-entries reflecting the comment hierarchy
- Status, priority, tags, and timestamps included
- Author information for each comment and reply

<<Export-Format>>
*** 13.2 Export Format [[#Export-Format][¶]]
:PROPERTIES:
:CUSTOM_ID: export-format
:CLASS: section
:END:
The exported org-mode structure preserves the nested reply hierarchy:

#+begin_src example-preformatted
,#+TITLE: Annotations for myfile.el
,#+DATE: 2025-06-19

,* TODO This needs review before release
:PROPERTIES:
:ID: thread-123456
:STATUS: open
:PRIORITY: high
:TAGS: review urgent
:CREATED: 2025-06-19T09:49:03
:AUTHOR: John Doe
:END:

The current implementation has potential security issues.

,** Reply by Jane Smith (2025-06-19T10:15:00)
I'll check this today and provide feedback.

,*** Reply by Bob Wilson (2025-06-19T10:17:00)
Ok!

,** Reply by Bob Wilson (2025-06-19T11:30:00)
Security audit completed. Issues documented in SECURITY.md
#+end_src

Nested replies appear at deeper heading levels (=**=, =***=, etc.),
making the conversation structure visible and foldable in org-mode.

<<Data-Structure-Evolution>>
*** 16.3 Data Structure Evolution [[#Data-Structure-Evolution][¶]]
:PROPERTIES:
:CUSTOM_ID: data-structure-evolution
:CLASS: section
:END:
Simply Annotate supports both legacy string annotations and new thread
structures:

- *Legacy format*: Simple strings are automatically converted to thread
  structures when edited
- *Thread format*: Complete data structures with metadata and
  conversation history
- *Level field*: Each annotation includes a =level= field (=file=,
  =defun=, or =line=)
- *Backward compatibility*: Old annotations without a =level= field
  default to =defun=. No migration required
- *Comment IDs*: Comments now include =id= and =parent-id= fields for
  nested threading. Legacy comments without these fields are
  automatically migrated on first access, with replies becoming children
  of the root comment
- *Migration*: Annotations are upgraded to thread format when modified.
  Use =M-x simply-annotate-migrate-threading= to migrate the entire
  database at once

- Check that all strings are properly quoted
- Verify required fields (id, comments) are present
- Use C-c C-k to cancel if unsure about changes

<<Data-Structure-Problems>>
**** 17.5.2 Data Structure Problems [[#Data-Structure-Problems][¶]]
:PROPERTIES:
:CUSTOM_ID: data-structure-problems
:CLASS: subsection
:END:
- Check timestamp format: use ISO format ="2025-06-29T10:30:00"=
- Ensure comment =type= field is ="comment"= or ="reply"=
- Verify =status= and =priority= values are from allowed lists
- Confirm =tags= field is a list, even if empty: =()=

<<Validation-Failures>>
**** 17.5.3 Validation Failures [[#Validation-Failures][¶]]
:PROPERTIES:
:CUSTOM_ID: validation-failures
:CLASS: subsection
:END:
- Thread structure must be an association list with required keys
- =comments= field must be a list of comment objects
- Each comment must have =author=, =timestamp=, =text=, and =type=
- Use the helpful comments in sexp editing mode as a guide

<<Performance-with-Large-Files>>
*** 17.6 Performance with Large Files [[#Performance-with-Large-Files][¶]]
:PROPERTIES:
:CUSTOM_ID: performance-with-large-files
:CLASS: section
:END:
Simply Annotate is designed to be lightweight:

- Annotations use lightweight overlays
- No performance impact on file operations
- Fringe mode generally has better performance than highlight mode
- Sexp editing may be slower with very large annotation structures
- Consider splitting very large annotation sets across multiple files

<<Integration-Issues>>
*** 17.7 Integration Issues [[#Integration-Issues][¶]]
:PROPERTIES:
:CUSTOM_ID: integration-issues

:END:
- Annotations work with most major modes
- Compatible with =lsp-mode=, =flycheck=, and other packages
- May need customization for exotic display modes
- Check for conflicting key bindings with C-h k
- Fringe indicators work well with line number modes
- Sexp editing works in any buffer that supports Emacs Lisp editing

<<Common-Error-Messages>>
*** 17.9 Common Error Messages [[#Common-Error-Messages][¶]]
:PROPERTIES:
:CUSTOM_ID: common-error-messages
:CLASS: section
:END:
- *"No annotation at point"* :: You're trying to edit or remove an
  annotation, but the cursor isn't positioned on annotated text. In
  fringe mode, make sure you're on the correct line.

- *"Annotation cancelled/removed"* :: You saved an empty annotation,
  which automatically removes it.

- *"No annotations in buffer"* :: The current buffer has no annotations
  to navigate or list.

- *"No annotations database found"* :: No annotation file exists yet.
  Create your first annotation to initialize the database.

- *"Invalid annotation format: must be a thread alist"* :: Sexp editing
  validation failed. Check the data structure format and required
  fields.

- *"Error parsing sexp"* :: The Elisp syntax in sexp editing mode is
  invalid. Check for balanced parentheses and proper quoting.

- *Database file errors* :: Check permissions and disk space for the
  storage file location.
