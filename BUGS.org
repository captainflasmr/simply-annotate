* Bugs/Improvements

** DONE =both= strategy wrote global annotations into the project database

Found during an elisp review (2026-05-31).  With =simply-annotate-database-strategy= set to =both=, every save copied all global annotations -- including absolute-path keys for unrelated projects -- into the project-local =.simply-annotations.el=.  Cause: =simply-annotate--update-database= loaded the *merged* project+global view (=--load-database=) and wrote it back to the project file.  Fixed so it reads/writes only the active store (=--database-path=); the two databases stay separate.  Covered by =sa-test-both-strategy-no-global-pollution=.  Already-polluted project files are not auto-cleaned (remove stray absolute-path entries by hand if needed).

** Important

*** TODO I1 A failed annotation-DB write can abort the user's buffer save

=simply-annotate.el= ~line 1955 (=--save-annotations=), ~870 (=--save-database=).  =--save-annotations= runs on =before-save-hook= and writes the DB via =with-temp-file= with no error guard.  If the DB path is unwritable (missing parent dir for a customised =simply-annotate-file=, read-only project root, full disk), the error propagates out of =before-save-hook= and aborts the user's actual file save.  Suggestion: wrap the DB write in =condition-case= and downgrade failures to a =message=/=warn= so the buffer still saves.

*** TODO I2 =Info-selection-hook= is added/removed globally and asymmetrically

=simply-annotate.el= ~line 5248 (add), ~5260 (remove).  Unlike the other mode hooks (all buffer-local with trailing =t=), =Info-selection-hook= is added/removed as a global hook on every mode toggle.  With the mode on in two buffers, disabling it in one removes the Info hook for all; it is also never cleaned up if the last enabled buffer is killed without toggling the mode off.  Suggestion: gate the global add/remove on whether any =simply-annotate-mode= buffer remains, or manage it once (first-enable / last-disable).

** Minor

*** TODO M1 Autoload cookies missing on a few user-facing commands

=simply-annotate.el= ~line 2405 (=simply-annotate-smart-action=), ~3135 (=simply-annotate-list=), ~5564 (=simply-annotate-dired-transient=).  Most entry points have =;;;###autoload=, but the primary =j= command, =simply-annotate-list=, and the dired transient do not.  They work via the loaded command-map/menu, but lose =M-x= discoverability before load.  Suggestion: add cookies for parity; the dired transient wants the same =;;;###autoload (autoload ...)= line that =simply-annotate-transient= has.

*** TODO M2 =(> (length x) 0)= for emptiness

=simply-annotate.el= ~line 1878, 4003 (and =(> (length ...) 1)= at 4013).  Walks the sequence just to test non-emptiness.  Suggestion: use truthiness / =consp= / =string-empty-p=.  Cosmetic, O(1).

*** TODO M3 =simply-annotate-show= silently no-ops with no annotation at point

=simply-annotate.el= ~line 2367.  =if-let= with no else returns nil silently when there is no overlay at point.  Suggestion: add an else branch with a short =message= (matching =simply-annotate-next= etc.).

*** TODO M4 =(nreverse (sort candidates ...))= to get descending order

=simply-annotate.el= ~line 4908.  Sorting ascending then reversing is wasteful/obscure.  Suggestion: sort directly with the reversed predicate (=string>= / swapped comparator).

** Optimisation

*** TODO O1 Quadratic card accumulation in kanban grouping

=simply-annotate.el= ~line 3865.  =(setcdr group (append (cdr group) (list card)))= appends to each group's tail for every annotation -> O(N^2) per status column.  Negligible for hundreds of annotations, scales poorly beyond.  Suggestion: =push= + =nreverse= once (the pattern already used at line 4553), or accumulate in a hash keyed by status.

*** TODO O2 =append=-in-loop for the kanban card map

=simply-annotate.el= ~line 4046.  =(setq all-card-map (append all-card-map (nth 1 result)))= inside the per-status loop is technically O(n^2); low real impact (~4 statuses).  Suggestion: =push=/=nconc= and reverse once.

*** TODO O3 =with-temp-buffer= + =fill-region= per line in word-wrap

=simply-annotate.el= ~line 896 (=--wrap-line=).  Allocates a temp buffer and runs =fill-region= per line of each comment on the render path.  Low priority -- =fill-region= is the right tool; only act if profiling shows it.  Suggestion: a reusable scratch buffer or a simple width splitter for the common case.

* Issues

Issues on the [[https://github.com/captainflasmr/simply-annotate/issues][GitHub Issues page]]

** DONE simply-annotate-file path cannot include =~= #10

Reported by pedro-nonfree (2026-04-12). Setting =(setq simply-annotate-file "~/path/to/simply-annotations.el")= did not work, while the fully expanded =/home/user/...= path did.

Fixed by normalising the path with =expand-file-name= at the choke points: =simply-annotate--read-db= now =expands its PATH argument (covering every read), =simply-annotate--database-path= returns an expanded path (covering every save), and the two direct =with-temp-file= / =delete-file= calls on =simply-annotate-file= in the migration commands were wrapped too.

** DONE allow using a different default state and priority than OPEN/NORMAL #9

Reported by pedro-nonfree (2026-04-12). New annotations were hardcoded to =open= status and =normal= priority, so a custom =(setq simply-annotate-thread-statuses '("inbox" "doing" "done" "closed"))= left new cards invisible in the kanban (the =open= column did not exist, and =--kanban-group-by-status= silently drops cards whose status has no column).

Fixed along the reporter's suggested lines:
- New helpers =simply-annotate--default-status= and =simply-annotate--default-priority= return =(car simply-annotate-thread-statuses)= / =(car simply-annotate-priority-levels)=.
- =simply-annotate--create-thread= now uses those helpers instead of literal =open= / =normal=.
- The default =simply-annotate-priority-levels= list was reordered to ='("normal" "low" "high" "critical")= so existing users keep =normal= as the default (list order is purely cosmetic — nothing sorts by it).
- Every "missing value" fallback (kanban grouping, project status counts, table views, header line, summary, status-cycling) now falls back to the configured default instead of literal =open= / =normal=, so legacy threads without a status field and plain-string annotations also land in the first column rather than vanishing.

Left as-is: the same-position sort heuristic (hardcodes =open=/=in-progress= as "active"; degrades to =string<= for custom statuses, only affects ordering) and the status→face =pcase= maps (unknown statuses get the default face).

** DONE usability feedback on keybindings (hydra? repeat-mode?) #8

Reported by pedro-nonfree (2026-04-12). Feedback that =C-c a= clashes with the org-mode manual's suggested binding for =org-agenda=; main ask: repeatedly typing the prefix is uncomfortable — would prefer a hydra-style or =repeat-mode= interface for chained commands.

Addressed via =repeat-mode= (the dependency-free, idiomatic option):
- New =simply-annotate-repeat-map= makes the repeatable commands (=next=, =previous=, =cycle-display-style=, =toggle-inline=, =cycle-tag-backward/forward=, =update-display-style=) chainable with bare keys (=n v ' / [ ] g=) once =repeat-mode= is on, with each command tagged via =(put cmd 'repeat-map ...)=.  Keys mirror the command-map; inert on Emacs 27.2.
- Documented in the Commentary alongside the existing =<prefix> SPC= transient menu (added in 2.2.0), which already gave a sticky/discoverable session.

Intentionally *not* done:
- *Hydra* -- would add a hard external dependency.  The reporter already has a working =defhydra= in their own config, which is the right place for it; =repeat-mode= + the transient cover the same ground built-in.
- *Changing the default prefix* -- the package already recommends =M-s= as primary and lets the user bind any prefix; =C-c a= is only the deferred-loading alternative in the quick-start.  The =org-agenda= clash is a per-user choice, not a package default to change.

** DONE annotation glitch in style =bar= when annotation is cancelled #7

Reported by pedro-nonfree (2026-04-12, against =simply-annotate-20260412.923= / commit b52fd63). Starting an annotation with =<prefix> j= and then cancelling (=C-g= or =C-c C-k=) correctly creates no annotation in the database, but left a leftover graphical artifact in the buffer.

Root cause: the text-based styles (=bar=, =bracket=, =bracket-text=) attach *auxiliary* alignment overlays (the =┃= / =┏= / =┗= indicators) stored as properties on the main overlay. The three draft-teardown paths (=simply-annotate--cancel-annotation=, =simply-annotate-cancel-edit=, =simply-annotate--cleanup-draft=) used a raw =delete-overlay= on the main draft overlay only, orphaning the indicator overlays. Fixed by routing all three through =simply-annotate--remove-overlay=, which runs the =--cleanup-*-overlays= helpers before deleting.

** DONE If an annotation is created outside of simply-annotate-mode, turning on the mode erases it #6

Fixed in 0.9.2. Mode activation now persists any existing annotation overlays to the database before clearing and reloading, so annotations created while the mode was off are never lost.

** DONE Minor mode activates keymap explicitly, leads to weird outcomes #5

Fixed in 0.9.3. Short-key bindings moved to =simply-annotate-command-map=, a separate prefix keymap. Enabling the minor mode no longer intercepts single-key presses during normal typing.

** DONE Right way to set key bindings to not interfere with existing bindings, like for Consult #4

Fixed in 0.9.3. =simply-annotate-command-map= can be bound to any prefix (e.g. =C-c a=) giving =C-c a j=, =C-c a r= etc. without interfering with Consult or other packages.

** DONE OPEN Support for org syntax in annotations #3

The annotation *editing* buffer now uses org-mode with org syntax support for comment editing.

** DONE Support for non-file buffers #2

Fixed in 0.9.1. Added =simply-annotate-file-key-functions= alist and =simply-annotate-file-key-function= buffer-local variable. Info-mode supported out of the box with per-node tracking.

** DONE More complex threading system #1

Fixed in 0.9.0. Full nested/hierarchical replies with comment =id= and =parent-id= fields, hierarchical =completing-read= for reply target and comment editing, tree-line display, and automatic legacy database migration.
